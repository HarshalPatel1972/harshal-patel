## 2024-05-24 - Do not implement multiple optimizations in a single PR
**Learning:** The prompt explicitly states to identify and implement ONE small performance improvement. Implementing multiple optimizations (e.g. Bot Regex and Redis Pipelining) in the same patch violates this directive and increases the risk of regressions or review failures. Additionally, modifying unverified third party or generic abstractions (like the `kv` vs `redis` object which use unverified return signatures like `[error, result][]` instead of a flat array) without exhaustive validation causes functional breakage.
**Action:** Strictly limit performance changes to exactly one verified optimization per PR, and never rewrite database bindings to use pipelining if the precise driver return structure (e.g. `ioredis` vs `@vercel/kv`) has not been fully inspected in the underlying implementation.
